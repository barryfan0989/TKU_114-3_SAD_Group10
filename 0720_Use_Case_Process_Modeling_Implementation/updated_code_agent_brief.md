# 更新後的程式碼代理實作任務書 (Updated Code Agent Brief)

本任務書是給程式碼代理（Code Agent）的實作規格書，詳細定義了第三個可操作切片的開發範圍、技術限制、不可自行決定的事項與完成驗收條件。

---

## 一、專案背景與目前狀態
本專案已完成前兩個可操作切片：
1. **第一切片**：完成社員預約表單登記、生成 RES- 唯一編號、財務後台模糊搜尋與一鍵確認收款、已繳費電子收據與社長即時統計。
2. **第二切片**：新增頂部模擬日期調整器、模擬 7 天預約自動逾期（BR-01）、現場快速切換繳費方案（FR-08）、社長逾期關懷名冊，以及電子收據大印與 CSS 列印排版最佳化。

本期目標是實作**第三個可操作切片**：**預約取消與收據作廢機制**，實現完整的合約與帳務生命週期控制。

---

## 二、本次實作範圍 (第三個可操作切片)

1.  **社員端自主取消預約 (UC-01 替代流程 A2)**：
    *   於一般社員我的預約面板中，當狀態為 `待繳費` 時，顯示「取消此筆預約」按鈕。
    *   點選彈出 confirm 確認框，點擊確認後將狀態更新為 `已取消`。
    *   釋放重複預約限制：當學員已被取消或作廢時，重新提交預約表單不再提示覆蓋，而是直接允許新增預約（利用逆向搜尋 `[...reservations].reverse().find` 來查找最新的有效預約狀態）。
2.  **財務端取消預約與作廢收據 (UC-04, UC-05)**：
    *   財務後台表格操作欄中：
        *   若狀態為 `待繳費`，除一鍵確認外，新增「取消」按鈕。點選後狀態更新為 `已取消`。
        *   若狀態為 `已繳費`，在綠色 badge 旁新增「作廢」按鈕。點選後彈出警告確認框，確認後狀態變更為 `已作廢`，並寫入 `voidTimestamp` 與 `voidHandler`（固定為財務幹部 林同學）。
3.  **社長看板作廢分析與雙名冊連動 (UC-07)**：
    *   社長指標網格中新增一個指標卡片：「已作廢/取消數」（顯示已取消與已作廢人數總和）。
    *   將已作廢與已取消的金額排除在社長「實收總額」與「待收預估」之外。
    *   下方未繳名冊旁新增「已作廢與已取消名冊」卡片，以條列式渲染出已取消或已作廢的社員、學號、方案、原金額、作廢時間與經辦人。
4.  **收據查驗作廢警告與列印截斷 (UC-06)**：
    *   收據查驗輸入學號查詢時：
        *   若狀態為 `已取消`，顯示已取消說明，不顯示收據。
        *   若狀態為 `已作廢`，渲染出紅色 dashed 邊框的作廢收據卡片，印上「收據已作廢 VOIDED」浮水印與圓形作廢章，且**不提供**「列印/下載」按鈕，防範行政核銷弊端。

---

## 三、技術與資料限制 (不可破壞)
1.  **CON-01 (單機版限制)**：限以單機 HTML/CSS/JS 開發，不引進任何後端伺服器 (Node/Express) 或真實資料庫。所有狀態以 LocalStorage 維持。
2.  **CON-02 (Notion 風格限制)**：堅持 Notion 簡約紙張質感設計，主要使用 `#37352f` 字體顏色與淡雅的 border 線條。新增按鈕與 badge 樣式需融入 Notion 質感（柔和灰色與紅色 dashed 邊框）。
3.  **防弊機制**：作廢操作為不可逆，且作廢收據需隱藏列印按鈕，保障評鑑嚴肅性。

---

## 四、不可自行決定的事項
1.  不可自行新增外部 SSO 登入或金流 API 連線代碼。
2.  不可自行移除或修改原本 7 天逾期的計算規則（BR-01）。
3.  作廢之經辦人固定為 `財務幹部 林同學`，不得隨機生成。

---

1.  修改的檔案名稱與變更內容。
2.  手動驗收測試案例操作步驟（MAT-01 ~ MAT-06）及通過情況。
3.  LocalStorage 資料結構的異動（如新增 `voidTimestamp` 與 `voidHandler`）。
