# 非功能性需求清單 (Non-functional Requirements)

以下為本專案的 5 筆非功能性需求（NFR），涵蓋可用性、效能、安全性與防弊、可維護性等多種品質屬性，並提供品質屬性情境描述。

| :--- | :--- | :--- | :--- | :--- | :--- | :--- | :--- |
| **NFR-01** | 可用性 (Usability) | 財務幹部使用手機或平板於社課現場收款 | 介面採用響應式網頁設計 (RWD)，排版適應不同螢幕寬度 | 身分切換欄、模糊搜尋框與「一鍵確認收款」按鈕不跑版，且點擊熱區符合手勢觸控規範 | 桌上型與行動端瀏覽器（如 Safari/Chrome） | INT-FIN-04 | 已確認 |

## 二、品質屬性情境 (Quality Attribute Scenarios)

我們為核心的可用性、效能與安全性需求，整理出 3 個品質屬性情境：

### 1. 可用性情境 (對應 NFR-05)
* **來源 (Source)**：一般社員
* **刺激 (Stimulus)**：送出缺少姓名欄位的繳費預約表單
* **環境 (Environment)**：一般操作情境
* **對象 (Artifact)**：預約登記表單
* **回應 (Response)**：阻止表單送出，在姓名輸入框旁顯示「學號與姓名皆為必填！」紅色訊息，並保留已選擇的方案（一年期 NT$ 900）
* **回應量測 (Response Measure)**：表單未建立新紀錄，畫面上顯示對應錯誤警告，且其他已填資料不消失

### 2. 效能情境 (對應 NFR-02)
* **來源 (Source)**：財務幹部
* **刺激 (Stimulus)**：在搜尋輸入框輸入學號的前四碼進行模糊搜尋
* **環境 (Environment)**：現場收現排隊高峰期
* **對象 (Artifact)**：財務後台預約資料表格
* **回應 (Response)**：系統執行模糊匹配，僅保留與前四碼相符的紀錄列，隱藏其他行
* **回應量測 (Response Measure)**：表格的過濾與重新渲染在 0.5 秒內完成

### 3. 安全性情境 (對應 NFR-03)
* **來源 (Source)**：未繳費社員或非法使用者
* **刺激 (Stimulus)**：嘗試在收據查驗頁面查詢「待繳費」或「已逾期」的預約收據
* **環境 (Environment)**：一般瀏覽器環境
* **對象 (Artifact)**：收據查驗模組
* **回應 (Response)**：系統拒絕生成電子收據憑證，並顯示醒目的狀態警告（例如「預約尚未完成繳款」或「已逾期作廢」）
* **回應量測 (Response Measure)**：不得產生包含印章與經辦欄位的電子憑證，只顯示錯誤警告
