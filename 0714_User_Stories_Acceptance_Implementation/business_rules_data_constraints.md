# 業務規則、資料需求與限制條件 (Business Rules, Data Requirements & Constraints)

本文件定義了 TkU 吉他社財務管理系統在業務邏輯、資料結構以及專案技術層面的規範與限制。

## 一、業務規則 (Business Rules)

| 規則編號 | 業務規則名稱 | 詳細業務邏輯說明 | 來源或推導依據 | 確認狀態 |
| :--- | :--- | :--- | :--- | :--- |
| **BR-01** | 7 天預約有效期限 | 繳費預約的有效期限為 7 天。如果預約單的 `dateCreated` 與目前系統日期的差距大於 7 天且仍未繳費（狀態為「待繳費」），系統應自動將其狀態標記為「已逾期」。已逾期的金額不計入社長看板的實收金額與待收預估中，且社員無法直接以該預約單查驗收據。 | INT-MEM-03 | 已確認 |
| **BR-02** | 一人一預約限制 | 系統強制每位社員（以學號唯一標識）在同一時間只能擁有一筆有效的預約或繳費記錄。若該學員已完成繳費（狀態為「已繳費」），則系統應拒絕其再次提交預約；若預約狀態為「待繳費」，社員再次提交預約時，系統應提示並允許覆蓋舊有待繳預約，重新生成預約編號並重新計算 7 天期限。 | 團隊推導（維護資料庫唯一性） | 已確認 |
| **BR-03** | 收付與退費規範 | 吉他社費僅提供半年期（NT$ 500）或一年期（NT$ 900）社費方案。現場收付一律採實體現金交易，不串接任何線上支付。且已點選「確認收款」之已繳費紀錄，在原型系統中排除退費或作廢功能，確保收據生成後資料的不可逆性與嚴肅性。 | INT-MEM-01, AI-GEN-01 | 已確認 |

## 二、資料需求 (Data Requirements)

| 資料編號 | 資料物件名稱 | 欄位定義與格式規範 | 推導依據 | 確認狀態 |
| :--- | :--- | :--- | :--- | :--- |
| **DR-01** | 預約主資料 (Reservation) | 包含以下欄位：<br>1. `reservationId`: 預約編號（格式：`RES-YYYYMMDD-三位隨機數`，主鍵）<br>2. `studentId`: 學員學號（必填，英文字母強制轉大寫）<br>3. `name`: 學員姓名（必填）<br>4. `scheme`: 方案類型（`full` 表示一年期，`half` 表示半年期）<br>5. `amount`: 應繳金額（500 或 900，單位：元）<br>6. `status`: 預約狀態（`待繳費` / `已繳費` / `已逾期`）<br>7. `dateCreated`: 預約建立日期（格式：`YYYY-MM-DD`） | FR-01, FR-02, BR-01 | 已確認 |
| **DR-02** | 電子收據資料 (Receipt) | 當預約確認收款後，應能產出包含以下資料的憑證：<br>1. `receiptId`: 收據編號（格式：將預約編號 `RES-` 替換為 `REC-`）<br>2. `club`: 繳費社團（淡江大學吉他社）<br>3. `memberInfo`: 學員姓名學號<br>4. `paymentDetails`: 付款方案與實收金額<br>5. `handler`: 經辦財務人員（固定為：`財務幹部 林同學`）<br>6. `timestamp`: 收款確認時間戳記（精確至分秒，格式：`YYYY/M/D 下午H:MM:SS`） | FR-05, FR-07, DOC-REG-01 | 已確認 |
| **DR-03** | 模擬系統參數 (SimParams) | 系統需於 LocalStorage 儲存並維護一個全局參數：<br>1. `simulatedDate`: 模擬系統日期（格式：`YYYY-MM-DD`）。頁面載入時預設為當前實際日期，並允許透過介面更改，以驅動 `BR-01` 的自動過期計算。 | BR-01 測試需求 | 已確認 |

## 三、限制條件 (Constraints)

| 限制編號 | 限制條件名稱 | 限制說明 | 來源依據 | 確認狀態 |
| :--- | :--- | :--- | :--- | :--- |
| **CON-01** | 單機 LocalStorage 限制 | 本原型系統限以單機版（HTML/CSS/JS）開發，排除任何後端伺服器部署與 SQL 資料庫串接。所有資料儲存與狀態持久化必須完全透過瀏覽器的 `LocalStorage` 模擬。 | CHA-SCOPE-01 | 已確認 |
| **CON-02** | 技術與 UI 框架限制 | 為了維持簡約 Notion 設計風格，禁止使用 React、Vue 等組件框架，也不得引入 Bootstrap、TailwindCSS 等外部 CSS 框架，介面排版與按鈕樣式必須純手寫 CSS 實現。僅允許引入外部 Google Fonts (Inter/Outfit) 與 FontAwesome 圖標字型庫。 | 團隊設計決策 (DESIGN-notion.md) | 已確認 |
| **CON-03** | 整合外部 API 限制 | 明確排除簡訊發送 API、真實線上金流 API（如 Line Pay、綠界 API 等），以及學校單一登入 (SSO) 認證。凡是需要外部連線的驗證功能一律以靜態 Toast 提示或本地端預載資料進行模擬。 | INT-MEM-04, AI-GEN-01 | 已確認 |
