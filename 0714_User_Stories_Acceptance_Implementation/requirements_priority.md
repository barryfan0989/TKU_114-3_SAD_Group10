# 需求優先順序表 (Requirements Prioritization)

本文件透過「必須／應該／可以／本次不做」方法（MoSCoW Method）與「價值投入矩陣」（Value-Effort Matrix）評估本專案所有需求的實作順序。

## 一、第一階段：MoSCoW 排序

| 需求/規則編號 | 需求名稱 | MoSCoW 等級 | 排序理由 | 本次是否實作 |
| :--- | :--- | :--- | :--- | :--- |
| **FR-01** | 線上選擇方案與預約 | **必須有 (Must)** | 系統核心輸入源，無此功能無法進行後續收款與統計。 | 是 |
| **FR-02** | 預約編號生成 | **必須有 (Must)** | 唯一編號是財務與社員核對憑證的關鍵鍵。 | 是 |
| **FR-03** | 模糊搜尋與篩選 | **必須有 (Must)** | 財務幹部於收款現場快速定位社員的必備工具，不可缺少。 | 是 |
| **FR-04** | 一鍵確認收款 | **必須有 (Must)** | 簡化現場收款操作，降低幹部現場工作負荷之核心功能。 | 是 |
| **FR-05** | 自動記錄經辦人與時間 | **必須有 (Must)** | 提供財務對帳與事後責任追溯（審計防弊）之法定/規章要求。 | 是 |
| **FR-06** | 社長統計儀表板 | **必須有 (Must)** | 提供社長決策社團經費與整體繳費進度的最高價值展現。 | 是 |
| **FR-07** | 符合評鑑的電子收據 | **必須有 (Must)** | 本專案解決紙本遺失與課外組財務評鑑合規的核心目標。 | 是 |
| **BR-01** | 7天逾期管理 | **必須有 (Must)** | 保障社團實收預估的精確度，防止陳舊未繳預約單虛胖財務報表。 | 是 |
| **CON-01** | LocalStorage 單機儲存 | **必須有 (Must)** | 技術限制，確保原型專案可在時程內完成交付之基礎。 | 是 |
| **FR-08** | 現場快速變更方案 (櫃檯) | **應該有 (Should)** | 解決訪談 Q1 现场繳費想臨時變更方案的需求，大幅降低幹部退回重建的阻力。 | **是 (第二切片實作)** |
| **DR-03 / Simulator** | 系統日期模擬器 | **應該有 (Should)** | 解決過期規則 (BR-01) 現場無法即時展示與測試的瓶頸。 | **是 (第二切片實作)** |
| **Care List** | 社長逾期關懷名冊 | **應該有 (Should)** | 解決訪談 Q2 逾期學員無法進行後續催繳與關懷的業務缺陷。 | **是 (第二切片實作)** |
| **Receipt Stamp** | 電子收據認證大印 | **可以有 (Could)** | 增加電子收據的真實感與美觀，讓課外組在財務審核時更有信心，屬於亮點功能。 | **是 (第二切片實作)** |
| **Print CSS** | 收據列印樣式最佳化 | **可以有 (Could)** | 提升電子收據的實用性，允許一鍵列印成乾淨紙本，屬於可用性提升。 | **是 (第二切片實作)** |
| **FR-08 (7/13)** | 自動 Email / 簡訊通知 | **本次不做 (Won't)** | 需要串接外部簡訊網關，且單機系統無伺服器，時程與技術風險過高。 | 否 |
| **FR-09 (7/13)** | 線上金流 API 整合 | **本次不做 (Won't)** | 涉及第三方支付帳號、資金安全及學校課外組法規限制，排除於本期。 | 否 |

## 二、第二階段：價值投入矩陣 (Value-Effort Matrix)

我們將本期規劃實作的功能繪製於價值投入矩陣中，藉此決定具體的開發順序：

```text
       高  ┌─────────────────────────┬─────────────────────────┐
           │ [第一優先：高價值、低投入]│ [第二優先：高價值、高投入]│
           │                         │                         │
           │ * FR-03 (模糊搜尋)      │ * FR-01 (預約表單)      │
           │ * FR-04 (一鍵確認)      │ * FR-06 (社長看板)      │
   價      │ * FR-05 (記錄經辦人/時間)│ * BR-01 (7天過期規則)   │
   值      │ * FR-08 (櫃檯方案變更)  │ * Simulator (時間模擬器)│
           │                         │ * FR-07 (電子收據)      │
           ├─────────────────────────┼─────────────────────────┤
           │ [第三優先：低價值、低投入]│ [第四優先：低價值、高投入]│
           │                         │                         │
           │ * Receipt Stamp (大印)  │ [已排除在範圍外]        │
           │ * Print CSS (列印最佳化)│   - 線上金流            │
           │                         │   - 簡訊與 Email 通知   │
       低  └─────────────────────────┴─────────────────────────┘
           低                       投入                       高
```

### 實作順序決策：
1. **階段 1 (第一個切片已完成)**：實作核心骨架（FR-01 至 FR-07, LocalStorage 基礎），快速打通「預約 - 收款 - 收據 - 社長看板」的流程。
2. **階段 2 (本次第二切片)**：
   - 實作高價值低投入的 **櫃檯現場變更方案 (FR-08)**，直接解決繳費排隊痛點。
   - 實作 **系統日期模擬器 (Simulator)**，讓高價值高投入的 **BR-01 逾期功能** 變得可以即時驗證。
   - 擴充社長看板以呈現 **逾期關懷名冊 (Care List)**。
   - 融入 **收據大印** 與 **列印 CSS 樣式** 滿足合規要求，雖然價值中等，但投入低，適合一併收尾。
