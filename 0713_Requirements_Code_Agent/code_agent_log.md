## 各階段交互 Prompt 與回應詳情紀錄 (Detailed Interaction History)

為了進一步展現疊代開發的深度，以下列出關鍵幾次交互的詳細 Prompt 與技術細節：

### 第一階段：架構規劃與可操作切片定義 (次數 1-2)
* **詳細 Prompts：**
  > 「你現在是 Antigravity AI 代理。我們要為吉他社開發社費繳費與電子收據原型系統。請遵守 `CON-01` 限制：使用單機 SPA 架構，僅能有 `index.html`、`style.css`、`app.js`，以 `LocalStorage` 作為資料儲存。系統需要涵蓋三個角色視角：一般社員、財務幹部、社長，以及收據查詢。請先規劃實作計畫，設計第一個可操作切片的資料庫欄位 schema 與檔案目錄結構。」
* **AI 回應與架構設計：**
  AI 規劃了單頁應用（SPA）的 DOM 切換架構，並定義了資料結構 schema：
  ```json
  {
    "id": "學號 (Primary Key)",
    "name": "姓名",
    "plan": "方案 (500 或 900)",
    "status": "狀態 (pending: 待繳費, paid: 已繳費, expired: 已逾期)",
    "bookingId": "預約編號 (RES-YYYYMMDD-XXX)",
    "receiptId": "收據編號 (REC-YYYYMMDD-XXX)",
    "amount": 500,
    "officer": "經辦人 (確認收款時寫入)",
    "timestamp": "收款時間戳記 (YYYY-MM-DD HH:mm:ss)",
    "bookingTime": "預約時間"
  }
  ```
  AI 同時提供了 SPA 的切換邏輯骨架：使用 `data-target` 屬性與 `hidden` 屬性控制不同角色視圖的顯示。

### 第二階段：核心業務邏輯與 LocalStorage 引擎實作 (次數 4-6)
* **詳細 Prompts：**
  > 「請實作 `app.js` 中的核心邏輯：
  > 1. 當 LocalStorage 為空時，初始化預載 20 筆測試資料。資料必須包含多樣狀態：12 筆已繳費、5 筆待繳費、3 筆已逾期，學號要符合 D112XXXX 格式。
  > 2. 實作社員表單送出邏輯：自動生成 `RES-` 格式編號，儲存至 LocalStorage。
  > 3. 實作財務搜尋邏輯：輸入學號或姓名時，即時模糊篩選。一鍵確認按鈕觸發時，將狀態改為 'paid'，自動寫入經辦人 '林同學' 與目前秒級時間戳記。」
* **人工代碼審查與修正 bug：**
  1. **事件監聽器丟失問題**：在財務搜尋框輸入文字後，表格會重新渲染，這導致之前綁定在「一鍵確認收款」按鈕上的 click 事件監聽器失效。
     * *解決方案*：人工重構為**事件委派 (Event Delegation)**，將點擊監聽器綁定在表格的 `<tbody>` 父容器上，透過 `event.target.closest('.btn-confirm')` 抓取被點擊的行，確保動態搜尋後按鈕功能依然正常。
  2. **學號大小寫敏感問題**：若預約時輸入小寫學號 `d1120017`，而財務搜尋或收據查詢時輸入大寫 `D1120017`，會無法匹配。
     * *解決方案*：在儲存與搜尋時，均使用 `.toUpperCase().trim()` 進行標準化處理。

### 第三階段：社長統計報表與 BR-01 邊界條件 (次數 7)
* **詳細 Prompts：**
  > 「實作社長看板的統計公式。社長需要即時看到：
  > 1. 已繳費率 = 已繳費人數 / 總人數 (預載資料 20 筆 + 新增資料)
  > 2. 已收總額 = 狀態為已繳費的方案金額加總
  > 3. 待收總額 = 狀態為待繳費的方案金額加總
  > 4. 請確保在財務點擊確認收款後，社長看板的數據能即時同步更新，不需重新整理網頁。」
* **人工修正核心邏輯 (BR-01 逾期排除)：**
  審查 AI 產出的計算代碼時，發現其寫法如下：
  ```javascript
  // 錯誤的 AI 原始寫法：待收總額包含了逾期項目，且已收總額未排除逾期狀態
  const totalPending = data.filter(item => item.status !== 'paid').reduce((sum, item) => sum + item.amount, 0);
  ```
  這違反了 `BR-01`（逾期預約的金額不得計入社長看板的實收與待收中，逾期應視為無效預約）。
  * *人工微調代碼*：
    ```javascript
    // 人工修正後的正確寫法：
    const totalPaid = data.filter(item => item.status === 'paid').reduce((sum, item) => sum + item.amount, 0);
    const totalPending = data.filter(item => item.status === 'pending').reduce((sum, item) => sum + item.amount, 0);
    const totalExpired = data.filter(item => item.status === 'expired').reduce((sum, item) => sum + item.amount, 0);
    // 逾期項目的金額單獨統計或直接排除，不計入實收與待收，維護財務數據的嚴謹性
    ```

### 第四階段：Notion 風格重構與行動端 UI 響應式優化 (次數 9-10)
* **詳細 Prompts：**
  > 「閱讀 `DESIGN-notion.md` 的 Notion 設計規範。我們需要將系統從暗色玻璃擬態風格，重構為極簡 Notion 紙質風格：
  > 1. 全域背景使用 `#f6f5f4` (Warm Paper)，內容卡片與導覽列使用純白 `#ffffff`，搭配 `1px solid #e6e6e6` (Hairline)。
  > 2. 文字顏色使用 `#37352f` (Ink Secondary)，標題使用 `#000000` (Ink)。
  > 3. 所有主要操作按鈕（如送出、確認、查詢）統一使用 Notion 藍 `#0075de` 作為背景。
  > 4. 表格採用 Notion 資料庫（Database）樣式，表頭為淺灰色 `#f6f5f4`，並以細線分隔。
  > 5. 確保在 iPhone 等移動端瀏覽時，表格不會撐破螢幕，按鈕大小合適。」
* **人工視覺與可用性微調：**
  - **表格響應式破版修正**：在小於 600px 的手機螢幕下，財務表格的「學號」、「姓名」、「方案」、「編號」、「狀態」、「操作」六個欄位會擠成一團，導致確認按鈕無法點擊。
    * *人工 CSS 修改*：為表格容器新增 `.table-container { width: 100%; overflow-x: auto; -webkit-overflow-scrolling: touch; }`，並將表格寬度固定在最小 `600px`，使手機用戶能順暢滑動檢視並精確點按「一鍵確認收款」按鈕（NFR-01 驗收通過）。
  - **收據列印樣式優化**：為電子收據卡片新增 CSS `@media print` 樣式，使社員在點擊瀏覽器列印時，只會列印出乾淨的收據憑證卡片，而自動隱藏頂部身分切換列與查詢框，大幅提升評鑑憑證的實用性。
