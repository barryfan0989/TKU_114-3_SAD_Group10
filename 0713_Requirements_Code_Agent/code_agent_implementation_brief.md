# 程式碼代理實作任務書 (Code Agent Implementation Brief)

本文件是給程式碼代理（Code Agent）的實作規格書，詳細定義了本次開發的範圍、技術限制、不可捏造事項與驗收條件。

---

## 一、專案背景
本專案為「社費預約繳費與電子收據管理系統」。目前社團多以人工收取現金及開立紙本收據，容易造成方案混淆、紙本收據遺失及對帳困難。本系統提供線上繳費預約、幹部現場模糊搜尋與一鍵確認、電子收據自動生成，以及社長即時統計看板。

---

## 二、第一個可操作切片 (First Working Slice)
本次實作只做：**「社員線上選擇方案送出預約，財務幹部在後台模糊搜尋並一鍵確認收款，系統隨即生成電子收據；社長後台即時更新繳費率統計與已繳/未繳名冊。」**

---

## 三、目標角色
1. **一般社員**：線上選擇方案並填寫學號與姓名送出預約；查詢並下載電子收據憑證；查看首頁狀態卡片。
2. **財務幹部**：透過搜尋框即時模糊搜尋社員；點擊「一鍵確認收款」按鈕確認收現並自動帶入經辦人與時間戳記。
3. **社長**：登入社長看板，即時查看繳費率（%）、已繳總額、未繳總額、已繳與未繳名單。

---

## 四、本期範圍內項目 (In-Scope)
1. **身分切換機制**：於網頁頂部提供一個切換欄，允許切換「一般社員蔡同學」、「財務幹部林同學」、「社長范同學」及「電子收據查詢」四種檢視。
2. **社員預約表單**：可輸入學號（必填）、姓名（必填）、選擇方案（半年期 500 元 / 一年期 900 元），點擊送出。
3. **預約編號生成**：送出後自動生成唯一預約編號（格式：`RES-YYYYMMDD-三位隨機數`）。
4. **狀態卡片提醒**：社員視圖首頁顯示最新預約狀態（如：待繳費、已繳費、已逾期）。
5. **財務模糊搜尋與一鍵確認**：支援輸入學號、姓名或預約編號進行即時模糊過濾，並提供一個「一鍵確認收款」按鈕。
6. **電子收據組件**：包含收據編號、社團名稱、姓名學號、方案金額、付款狀態（已繳費）、經辦人（固定為財務幹部林同學）與精確到分秒的確認時間戳記。
7. **社長儀表板**：顯示已收金額、待收金額、缴費率（以 20 筆預載資料 + 新增資料即時計算）、已繳與未繳費社員清冊。
8. **20 筆預載測試資料**：預先載入包含「已繳費」、「待繳費」、「已逾期」等狀態的 20 筆社員資料（LocalStorage 未初始化時自動載入）。

---

## 五、本期範圍外項目 (Out-of-Scope)
1. **真實後端伺服器與資料庫**：全部資料儲存與邏輯均在前端使用 LocalStorage 模擬。
2. **正式登入與密碼驗證**：僅使用頂部身份切換按鈕進行模擬切換。
3. **線上金流串接**：排除 Line Pay、信用卡或銀行 API，僅處理現場現金收付的狀態確認。
4. **外部自動化通知**：不串接自動發送 Email 或簡訊催繳服務，通知僅限於系統內首頁狀態卡片的更新。
5. **退費/退會審核流程**：本期不支援退費與取消已確認繳費之操作。

## 六、需求與驗收條件 (Acceptance Criteria)

| 需求編號 | 需求描述 | 驗收條件 (Acceptance Criteria) |
| :--- | :--- | :--- |
| **FR-01** | 線上選擇方案與預約 | 1. 必填欄位空白時，點擊預約應顯示錯誤提示且無法送出。<br>2. 選擇「一年期」送出後，待繳金額應顯示 900 元。 |
| **FR-02** | 預約編號生成 | 送出預約後，畫面上必須顯示以 `RES-` 開頭的唯一預約編號。 |
| **FR-03** | 模糊搜尋 | 在財務後台搜尋框輸入部分學號或姓名，列表應在 0.5 秒內即時過濾，不符合的資料必須隱藏。 |
| **FR-04** | 一鍵確認收款 | 點擊待繳費社員旁的「一鍵確認收款」按鈕後，該筆資料狀態應立即變為「已繳費」，按鈕隨即消失或變為不可點擊。 |
| **FR-05** | 記錄經辦人與時間 | 點擊一鍵確認後，該筆資料中應被寫入確認時間戳記，以及經辦人「林同學」。 |
| **FR-06** | 社長統計儀表板 | 社長看板的已繳費率（%）、已收總額與未繳名冊，在財務確認收款後必須即時重新計算並更新，無須手動重整網頁。 |
| **FR-07** | 電子收據查驗 | 1. 已繳費社員可在收據頁面輸入學號查詢並顯示收據。<br>2. 收據上必須清楚呈現「收據編號」、「姓名學號」、「已付金額」、「經辦人林同學」及「付款時間」。 |
| **BR-01** | 逾期預約管理 | 預載資料中應有狀態為「已逾期」的項目，且逾期項目的金額不得計入社長看板的已收總額中。 |
| **NFR-01** | 行動裝置適配 | 使用 Chrome 開發者工具模擬手機（如 iPhone）尺寸，搜尋框與一鍵確認按鈕不可跑版，必須易於單手點擊。 |

## 七、技術限制
1. **單一網頁應用 (SPA)**：僅使用單一 `index.html`，搭配 `style.css` 及 `app.js`。
2. **純前端技術**：不可使用 React、Vue、Angular、Vite 或 Next.js。限用原生 HTML5、Vanilla CSS 及 Vanilla JavaScript。
3. **資料持久化**：使用瀏覽器的 `LocalStorage` 儲存所有預載資料與新增預約資料。
4. **無外部相依性**：不可使用 npm 套件、外部資料庫或正式 API。

---

## 八、程式碼代理不可自行決定的事項
1. **不可自行新增「維修廠商」或「課外組承辦人」等不存在的角色視圖。**
2. **不可自行變更社費方案與金額（半年期 500 元，一年期 900 元）。**
3. **不可自行加入真實註冊與真實密碼輸入功能。**
4. **不可自行引入外部第三方 CSS/JS 框架（如 Bootstrap, TailwindCSS 等，除非小組明示）。**

---

## 九、若代理想加入，必須拒絕的功能

| 想加入的功能 | 拒絕理由 | 對應範圍外項目 |
| :--- | :--- | :--- |
| **自動 Email / 簡訊通知** | 增加外部 API 串接技術難度與測試成本。 | 五.4 外部自動化通知 |

| **Excel 報表匯出功能** | 雖然實用，但超出本次第一個可操作切片的核心展示範圍。 | 五.1/五.5 範圍限制 |

## 十、交給程式碼代理的正式任務文字 (用於啟動 Subagent)
```text
請依照以下系統規格，在 `0713_Requirements_Code_Agent/first_working_slice/` 目錄中，建立第一個可操作切片的單頁網頁原型（包含 `index.html`, `style.css` 及 `app.js`）：

1. 專案背景：
   面向一般社員、財務幹部與社長之社費預約繳費、一鍵確認與電子收據管理系統。

2. 技術限制：
   使用原生 HTML5、Vanilla CSS（手寫設計美學，不可用框架）及 Vanilla JS。使用 LocalStorage 進行資料模擬與狀態持久化，未初始化時需預載 20 筆具備已繳、未繳、逾期狀態之社員測試資料。

3. 第一個可操作切片流程：
   - 頂部有身分切換欄，可切換「一般社員蔡同學」、「財務幹部林同學」、「社長范同學」及「電子收據查詢」視窗。
   - 一般社員視圖：可線上選擇方案（半年期 500 元 / 一年期 900 元）並填寫學號姓名送出，生成唯一預約編號（`RES-YYYYMMDD-三位隨機數`），並顯示待繳卡片。
   - 財務幹部視圖：提供即時模糊搜尋，可輸入學號/姓名/預約編號過濾，對待付款項目提供「一鍵確認收款」按鈕。點擊後狀態改為「已繳費」，並記錄時間戳記與經辦人「林同學」。
   - 社長視圖：即時計算並顯示繳費率、已收總額、待收總額與已繳/未繳社員清冊。
   - 電子收據查詢視圖：可輸入學號查詢已繳費項目的電子收據（含收據編號、姓名學號、付款方案、已繳金額、確認時間、經辦人與付款成功狀態）。

4. 驗收條件：
   - 社員送出預約後，財務後台模糊搜尋能即時搜到該筆待繳預約。
   - 財務點擊「一鍵確認收款」後，社長看板的繳費率與實收金額需即時更新。
   - 社員能以學號查出包含「經辦人：林同學」與精確時間戳記的收據。
   - 介面須具備高質感（Notion 簡約紙質風格設計，以細邊框 hairline 及微陰影分層，色彩克制並只在主要動作使用 Notion 藍）。

請先規劃檔案結構與實作計畫，然後建立檔案，並在完成後列出已實現的需求編號。
```
